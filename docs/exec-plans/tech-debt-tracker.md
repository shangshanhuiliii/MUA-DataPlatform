### 批次筛选仅支持前 100 个批次

- 来源: `github/feature/search-page-jump` 分支 Code Review
- 状态: 已识别，暂缓处理
- 当前决策: 暂时接受，作为技术债跟踪
- 接受原因: 当前批次数量有限，短期内不会影响实际使用
- 问题描述: 任务/录制选择弹窗中的“按批次筛选”依赖一次性拉取批次列表，当前实现最多只能拿到前 100 个批次；当批次数超过 100 时，后续批次不会出现在筛选下拉框中
- 影响范围: 任务选择弹窗、录制选择弹窗
- 触发条件: 系统中的批次数量大于 100
- 建议处理方向:
  - 将批次筛选改为支持搜索的远程下拉
  - 或为批次列表增加分页/增量加载能力
  - 避免前端依赖一次性全量拉取批次

### 选择弹窗的权限状态来源仍然分散

- 来源: `github/feature/search-page-jump` 分支 Code Review
- 状态: 已识别，暂缓处理
- 当前决策: 暂时接受，作为技术债跟踪
- 当前结论: 现阶段手工测试未复现实际功能问题
- 问题描述: 任务/录制选择弹窗当前仍在组件内部自行判断管理员权限，未统一复用上层的当前用户状态；这使权限相关 UI 逻辑分散在多个组件中，增加后续维护和排查成本
- 影响范围: 任务选择弹窗、录制选择弹窗、任务录制页权限相关 UI
- 风险说明: 当前风险主要是架构复杂度和后续演进成本，而不是已确认的线上故障
- 建议处理方向:
  - 统一由上层维护当前用户信息和权限状态
  - 为前端提供单一的身份/权限读取入口
  - 由调用方将权限信息传入弹窗，避免弹窗内部重复做身份判断

## 2026-03-26

### 录制异常扫描对非对象 task-info.yaml 不够健壮

- 来源: `docs/plans/2026-03-25-recording-repair-code-review.md`
- 状态: 已识别，暂缓处理
- 当前决策: 本轮不修实现，作为技术债跟踪
- 问题描述: `recording_exception_service.load_task_info_summary()` 默认把 `task-info.yaml` 解析结果当作字典使用；如果历史文件能被 YAML 解析但结果是列表、字符串或数字，异常扫描接口 `/api/recordings/exceptions` 会因为对非对象调用 `.get()` 而报错
- 影响范围: 录制异常管理页、管理员历史录制修复链路
- 触发条件: 历史录制目录中的 `task-info.yaml` 内容格式异常但仍是合法 YAML
- 建议处理方向:
  - 在读取 `task-info.yaml` 后先校验类型，仅在 `dict` 情况下读取字段
  - 对非对象 YAML 回退为“task-info 存在但不可结构化读取”的容错分支
  - 为列表/字符串/数字等坏历史数据补充异常扫描测试

### 录制异常修复弹窗只加载前 100 个用户

- 来源: `2026-03-26` 最新 Code Review
- 状态: 已识别，暂缓处理
- 当前决策: 本轮不修实现，作为技术债跟踪
- 问题描述: 录制异常修复弹窗前端调用 `GET /api/users` 时未传 `limit`，而后端默认 `limit=100`；当用户数量超过 100 时，下拉框会静默缺失后续用户，管理员无法为这些录制选择真实录制人
- 影响范围: 录制异常管理页、管理员手工修复录制归属流程
- 触发条件: 系统用户数量大于 100，且待修复录制的真实录制人不在前 100 个返回结果中
- 建议处理方向:
  - 将修复弹窗的用户选择改成支持搜索的远程下拉
  - 或显式提高后台管理场景的拉取上限，并补齐“结果被截断”的前端提示
  - 避免后台修复流程依赖一次性全量拉取用户列表

### 录制链路 `dataset` 命名残留未收敛为 `recording`

- 来源: `2026-03-26` 当前修复过程中的追加评审意见
- 状态: 已识别，暂缓处理
- 当前决策: 因涉及前后端 websocket 消息字段和录制线程参数，本轮不做半套改名，作为技术债跟踪
- 问题描述: 虽然业务语义已经从 `dataset` 收敛到 `recording`，但 `task_record_service.py`、`recording_worker_thread.py`、`task-recorder.js` 等链路仍在使用 `dataset` 命名；当前不仅有局部变量残留，还包括 websocket 消息字段 `message.dataset`、会话属性 `self.dataset`、以及录制线程构造参数 `dataset`。如果只改后端或只改前端，会导致录制启动、录制就绪回传、追加录制等链路字段不一致
- 影响范围: 任务录制 websocket 链路、录制线程初始化、前端录制状态同步、追加录制消息格式
- 触发条件: 继续在局部文件中零散推进 `dataset -> recording` 改名，而未同步更新前后端消息协议
- 建议处理方向:
  - 一次性梳理 websocket 请求/响应字段，统一把外部协议字段从 `dataset` 改为 `recording`
  - 同步收敛 `RecordingSession`、`RecordingWorkerThread` 等内部命名，避免同一概念多套叫法
  - 为 `start_recording`、`recording_ready`、追加录制等关键链路补前后端联调测试，防止改名过程中协议断裂

### recording owner / access 后端校验仍未完全收口

- 来源: `2026-04-03` 最新 Code Review
- 状态: 已识别，暂缓处理
- 当前决策: 本轮不改实现，先作为技术债跟踪
- 问题描述: recording 访问控制当前仍部分依赖前端约束；`set_current_recording()` 对前端传入的已有 `directory_name` 仅校验目录存在，不校验是否属于当前用户；`append_data` 分支同样只校验目录存在，未统一执行 ownership / access 校验
- 影响范围: 当前录制切换、追加录制链路、后端权限边界
- 风险说明: 当前实际风险低于最初 workspace 泄漏问题，但仍属于后端 trust client 的权限缺口，不应作为长期边界
- 建议处理方向:
  - 为 `set_current_recording()` 补齐 recording owner / access 校验
  - 为 `append_data` 分支统一复用同一套 recording 访问校验
  - 为普通用户 / 管理员 / 非属主场景补后端权限测试

### refresh token cookie 尚未按环境启用 Secure

- 来源: `2026-04-03` 最新 Code Review
- 状态: 已识别，暂缓处理
- 当前决策: 本轮不改 cookie 策略，先作为技术债跟踪
- 问题描述: 当前 refresh token cookie 已承载长期续期能力，但 `_set_refresh_cookie()` 尚未设置 `Secure`，在非 HTTPS 链路上会以明文发送
- 影响范围: 浏览器 refresh cookie 传输安全、生产环境续期链路
- 风险说明: 本地开发场景需要兼容 HTTP，但生产环境不应长期保持无 `Secure` 的默认配置
- 建议处理方向:
  - 按环境或显式配置补齐 `Secure` 策略，例如 `secure=not DEBUG`
  - 为生产部署提供强制开启 `Secure` 的配置项
  - 补充部署文档，明确 HTTPS 与 cookie 策略要求
